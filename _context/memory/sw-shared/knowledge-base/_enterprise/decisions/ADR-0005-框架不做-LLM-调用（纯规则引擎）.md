# ADR-0005: 框架不做 LLM 调用（纯规则引擎）

**状态:** `accepted`
**日期:** `2026-05-04`
**决策者:** `hw-controller`
**Scope:** enterprise

## 背景

harness-framework 是多 Agent 协作的编排引擎。在设计初期，需要考虑框架是否应该具备智能决策能力（如通过 LLM 分析任务优先级、智能分配 Agent、自动优化 DAG 拓扑等）。框架本身是一个 Python 守护进程，三条线程分别运行 Aggregator、Watchdog 和 WebAPI。

## 决策

**我们决定:** harness-framework 的核心引擎不做任何 LLM 调用，完全作为纯规则引擎运行。所有智能决策由 Agent 层（multiagents）完成，框架只负责状态管理和流程控制。

## 理由

1. 框架作为"基础设施层"，其行为必须是 100% 可预测和可测试的——状态机转换规则确定，CAS 竞争结果确定，超时回收逻辑确定
2. LLM 调用的不确定性（输出不稳定、延迟高、成本不可控）与框架的确定性要求矛盾
3. 框架以单进程多线程方式运行（Aggregator 每 5 秒轮询，Watchdog 每 30 秒轮询），如果嵌入 LLM 调用会阻塞轮询循环
4. 职责分离清晰：框架 = 流程控制（确定性的），Agent = 智能决策（非确定性的），测试责任边界明确
5. 框架的性能瓶颈在 I/O（Consul HTTP 请求），LLM 调用的延迟会放大框架的响应时间

## 考虑的替代方案

| 方案 | 优点 | 缺点 | 为什么不选 |
|------|------|------|-----------|
| 框架调用 LLM 分析任务优先级 | 可以动态调整优先级，优化 DAG 执行顺序 | LLM 调用延迟（秒级）阻塞框架轮询，不可预测 | 框架的轮询间隔是秒级（5s），LLM 调用会破坏时序确定性 |
| 框架调用 LLM 生成失败原因和建议 | 提供更有价值的错误信息 | 框架中需要集成 LLM 客户端库，增加依赖和复杂度 | 失败分析和建议是 Agent 的职责（Watchdog 只需写 error_message） |
| 框架调用 LLM 做智能 Agent 分配 | 可以匹配 Agent 能力和任务类型 | 需要框架维护 Agent 能力知识库，引入额外的 LLM 成本 | 已经确定 Agent 主动抢占模式，不需要框架做分配 |

## 后果

### 正面
- 框架行为完全确定：给定相同的输入，永远产生相同的输出
- 测试简单：纯逻辑验证，不需要 mock LLM 接口，不需要处理 LLM 返回的随机性
- 性能稳定：框架的延迟完全取决于 Consul 响应时间，不会出现 LLM 引起的尖峰延迟
- 成本可控：框架运行不消耗 LLM Token

### 负面
- 框架不能提供智能层面的增值功能（如自动分析失败模式、优化调度策略）
- 某些跨任务的协调逻辑需要人工编写规则（如复杂的条件依赖）
- LLM 的功能被限制在 Agent 层，Agent 需要自己查询框架的状态来做出决策
- 缺少"框架级"的智能，Agent 各自使用 LLM 可能导致整体效率不是最优
