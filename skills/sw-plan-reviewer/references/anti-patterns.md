# AntiPatterns: 反模式目录

以下模式在审查中**经常出现，但不应作为 REJECT 的理由**。每个反模式附有说明、示例和正确处理方式。

---

## 反模式 1: "可以更清晰"建议

### 描述
指出某个任务的描述"可以更清晰"、"可以更详细"但开发者仍然能理解要做什么。

### 为什么不是阻塞问题
开发者拥有推断能力。不需要像法律合同一样精确的计划文档。模糊但可开始的任务不是一个阻塞问题。

### 示例（不阻塞）
```
任务 3: "改进错误处理"
审查者评论: "任务 3 可以更清晰，应该说明具体哪里需要改进"
→ 这是 NOT A BLOCKER。开发者理解"改进错误处理"意味着在现有代码中添加 catch/error handler。
```

### 正确处理方式
- **OKAY。** 不将此列为问题。开发者可以理解并开始工作。
- 如果错误处理是任务的核心目的，且当前完全没有错误处理的入口 → 这才是 REJECT。

## 反模式 2: "考虑添加验收标准"

### 描述
指出某个任务缺少显式的验收标准（acceptance criteria），但 QA 场景中已包含验证步骤。

### 为什么不是阻塞问题
验收标准是锦上添花，不影响可执行性。QA 场景已经提供了可验证的检查点。

### 示例（不阻塞）
```
任务 4: "实现用户登录 API"
QA 场景: curl -X POST /api/auth/login -d '{"username":"test","password":"pass"}' → 预期 200 + token
审查者评论: "任务 4 考虑添加明确的验收标准"
→ NOT A BLOCKER。QA 场景已经清晰描述了验证方式。
```

### 正确处理方式
- **OKAY。** QA 场景本身就起到了验收标准的作用。
- 如果整个计划都没有 QA 场景或验收标准 → 这才是 REJECT。

---

## 反模式 3: "方案可能不是最优"

### 描述
认为任务的实现方案有更优解，但当前方案可以工作。

### 为什么不是阻塞问题
审查者的工作是验证可执行性，不是评估实现方案的优化程度。你怎么做这件事与本审查无关。

### 示例（不阻塞）
```
任务 5: "使用简单的 for 循环遍历列表并过滤"
审查者想法: 应该用 filter() 或列表推导式，性能更好
→ 无论 for 循环还是 filter()，任务都是可执行的。NOT A BLOCKER。
```

### 正确处理方式
- **OKAY。** 不将此列为问题。
- 如果方案本身不可能实现（如"在 JavaScript 中直接调用 Python 模块"）→ 这才是 REJECT。

---

## 反模式 4: "缺少边缘情况文档"

### 描述
指出某个任务没有覆盖所有边缘情况（网络故障、空数据、并发等），但这些不是任务的主流程。

### 为什么不是阻塞问题
除非边缘情况是任务的主要目的，否则开发者可以在实现过程中处理常见边缘情况。文档中是否列出边缘情况不影响可执行性。

### 示例（不阻塞）
```
任务 2: "实现用户注册端点"
审查者评论: "缺少对邮箱已注册、用户名已占用、密码过短等边缘情况的说明"
→ NOT A BLOCKER。这些是注册功能的标准边缘情况，开发者知道需要处理。
```

### 例外 — 当边缘情况变成阻塞问题
```
任务: "实现支付处理端点"
边缘: 没有提到并发支付或重复支付的处理
→ 如果支付系统的核心风险就是并发和幂等，缺少这些才算 BLOCKER。
但即使如此，除非任务完全没有提及"支付"的具体操作，否则这仍然是开发者知道需要处理的。
```

### 正确处理方式
- **OKAY，即使边缘情况没有明确列出。**
- 仅当**主流程本身缺失**描述时才 REJECT。

## 反模式 5: "拒绝因为你做的方式不同"

### 描述
审查者不喜欢任务的实现风格、命名、组织结构或技术选择，但此任务在技术上是可行的。

### 为什么不是阻塞问题
不同的开发者有不同的风格。只要任务在技术上是可执行的，风格偏好不能构成阻塞理由。

### 示例（不阻塞）
```
任务 1: "把组件放在 src/components/common/ 下"
审查者想法: 应该放在 src/shared/ui/ ，common 这种命名太模糊了
→ 两个路径都是存在的目录选择。无论哪个路径任务都是可执行的。NOT A BLOCKER。
```

### 正确处理方式
- **OKAY。** 绝不以个人偏好为由拒绝。
- 仅当目标路径与计划其他部分冲突时 → 这才作为矛盾阻塞。

---

## 反模式 6: "缺少架构层次的文档"

### 描述
指出计划没有描述系统架构、数据流或模块间的完整通信协议。

### 为什么不是阻塞问题
本审查检查的是工作计划的**可执行性**，不是架构文档的完整性。架构设计应该有独立的文档或设计评审来处理。

### 示例（不阻塞）
```
审查者评论: "计划没有说明 auth service 和 user service 之间的数据流是什么"
→ 如果每个服务中各自的任务都描述清楚了要做什么，缺少高层数据流图不影响单个任务的可执行性。
NOT A BLOCKER。
```

### 正确处理方式
- **OKAY。** 架构设计不是本审查的范围。
- 除非两个任务直接矛盾（如一个说 A→B，另一个说 B→A），才作为矛盾阻塞。

---

## 反模式 7: "列出了超过 3 个问题"

### 描述
审查者找到了 4+ 个他们认为是问题的点并全部列出。

### 为什么这是反模式
一次列出太多问题会让计划作者不堪重负，降低每个问题的被处理概率。只列最关键的 3 个，其余的可以等作者修复这 3 个后再发现。

### 示例（不这样做）
```
审查者输出:
Blocking Issues:
1. 引用文件 X 不存在
2. 任务 Y 没有文件路径
3. 任务 Z 与任务 W 矛盾
4. 缺少错误处理
5. QA 场景没有预期结果
6. 引用文件 X 的行号不对
7. 接口签名不完整
→ 作者看到 7 个问题会感到不堪重负；其中一些问题不是阻塞问题
```

### 正确处理方式
只列关键的 3 个阻塞问题。如果有更多问题，选取：
1. 会导致工作完全停滞的问题（最高优先级）
2. 最多任务受影响的问题
3. 最早出现问题的影响范围（计划前面部分的问题优先）

## 反模式 8: "以'进一步细节化'为由拒绝"

### 描述
要求计划包含更多细节，但当前细节水平已经足够开发者开始工作。

### 为什么不是阻塞问题
计划的目的是指导，不是替代。计划不是代码本身——计划是指向代码的路标。路标不需要写满 GPS 坐标。

### 示例（不阻塞）
```
任务: "在 user_service 中实现 getUser 方法，返回 User 对象"
审查者评论: "应该列出 User 对象的完整字段定义"
→ NOT A BLOCKER。开发者知道要在 user_service 中实现 getUser，Task 描述已经足够。
如果 User 对象定义需要与其他服务对齐，这应该在其他文档中（如 API 设计文档），不是每个任务都要重复一遍。
```

### 正确处理方式
- **OKAY。** 不因细节不够多而拒绝。
- 仅当连"在哪里做什么"都不清楚时才 REJECT。

---

## 反模式 9: "缺少性能考量"

### 描述
指出计划没有讨论性能约束、数据库优化或缓存策略。

### 为什么不是阻塞问题
性能审查是 `sw-reviewer-performance` 的职责，不是本审查的范围。除非计划中明确提到了性能要求但实现方式直接违背该要求。

### 正确处理方式
- **OKAY。** 性能不属于本审查范围。

---

## 快速记忆口诀

审查前问自己：
1. 这个问题会**完全阻止**开发者开始工作吗？ 不 → 批准
2. 这个问题是我**个人偏好**吗？ 是 → 批准
3. 这是**第 4+ 个**问题吗？ 是 → 选前 3 个，其余不提
4. 这个问题属于**代码质量/性能/安全**范畴吗？ 是 → 不属于本审查，批准
