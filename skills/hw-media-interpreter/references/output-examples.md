# 输出示例 (Output Examples)

展示 hw-media-interpreter 在各个场景下的输入与正确输出格式。核心原则：**直接返回提取的信息，无前言、无后语、无多余 token。**

---

## 示例 1: PDF -- 提取特定章节内容

### 输入 (Caller)

```
file_path: /path/to/system-design.pdf
goal: 提取第2章"系统架构"的所有内容
```

### 正确输出

```
## 第2章 系统架构

### 2.1 总体架构

本系统采用微服务架构，分为接入层、业务层、数据层三层：

- 接入层：API Gateway 负责统一入口、认证鉴权、流量控制
- 业务层：多个独立部署的微服务，通过 gRPC 通信
- 数据层：MySQL 主从集群 + Redis 缓存 + Elasticsearch 搜索

### 2.2 技术选型

| 组件 | 技术 | 版本 | 选型理由 |
|------|------|------|----------|
| API网关 | Kong | 3.x | 高性能、插件丰富 |
| 服务框架 | Spring Boot | 3.2 | 团队熟悉、生态成熟 |
| 数据库 | MySQL | 8.0 | 强一致性需求 |
| 缓存 | Redis | 7.2 | 高性能、数据结构丰富 |
| 消息队列 | Kafka | 3.6 | 高吞吐、持久化 |

### 2.3 部署架构

生产环境部署于 Kubernetes 集群，共 12 个 Pod：
- API Gateway: 3 副本
- 业务服务: 6 副本（每个服务至少 2 副本）
- 基础设施: 3 节点（MySQL/Redis/Kafka 各1）
```

### 错误输出（反例）

```
好的，我来帮你提取第2章的内容。让我先读取这个PDF文件。

[读取结果...]

根据分析，第2章"系统架构"的内容如下：

## 第2章 系统架构
...

以上就是第2章的全部内容。如果还需要提取其他章节，请随时告诉我。
```

**问题：** 有前言（"好的，我来帮你..."）和后语（"以上就是..."），浪费了上下文 token。

```
file_path: /path/to/api-spec.pdf
goal: 提取文档中所有的API参数表格
```

## 示例 3: 架构图 -- 描述组件与关系

### 输入 (Caller)

```
file_path: /path/to/microservices-arch.png
goal: 描述这个微服务架构图中的所有组件、分层和调用关系
```

### 正确输出

```
架构层级（从上到下）：

[用户端]
- Web App (React): 浏览器端 SPA
- Mobile App (React Native): iOS/Android

[接入层]
- API Gateway (Kong): 统一入口
  - 职责：路由、认证(JWT)、限流、日志
- Load Balancer (Nginx): 前置负载均衡

[业务服务层]
- User Service: 用户管理（注册、登录、权限）
  端口: 8001 | 协议: gRPC（内部）/ REST（通过Gateway）
- Order Service: 订单管理（创建、支付、退款）
  端口: 8002 | 协议: gRPC（内部）
- Product Service: 商品管理（CRUD、库存）
  端口: 8003 | 协议: gRPC（内部）
- Notification Service: 消息推送（邮件、短信、站内信）
  端口: 8004 | 消费者模式

[消息中间件]
- Kafka: 订单事件、通知事件

[数据层]
- MySQL (主从): User/Order/Product 各自独立数据库
- Redis Cluster: 会话缓存、热点数据
- Elasticsearch: 商品搜索

[基础设施]
- Kubernetes: 容器编排
- Prometheus + Grafana: 监控告警
- ELK: 日志收集分析

关键调用关系：
- 客户端 --HTTPS--> API Gateway --内部路由--> 各业务服务
- User Service --gRPC--> Order Service (验证用户权限)
- Order Service --Kafka--> Notification Service (订单状态变更通知)
- Order Service --gRPC--> Product Service (扣减库存)
- 所有服务 --写入--> 各自的 MySQL 数据库
- 所有服务 --读取--> Redis (缓存)
- Product Service --同步--> Elasticsearch (搜索索引)

外部集成：
- 支付网关 (支付宝/微信支付): Order Service 通过回调接口交互
- 邮件服务 (SMTP): Notification Service 通过 SMTP 发送
```

## 示例 4: 流程图 -- 描述业务流程

### 输入 (Caller)

```
file_path: /path/to/order-flow.png
goal: 描述这个订单处理流程的完整步骤和决策分支
```

### 正确输出

```
流程名称：订单处理流程

起点：用户提交订单
终点：订单完成/取消

步骤与决策：

1. 用户提交订单
   -> 系统验证库存

2. [决策] 库存是否充足？
   - 否 -> 提示用户"库存不足" -> 终点: 订单取消
   - 是 -> 进入步骤3

3. 锁定库存
   -> 创建订单记录（状态:待支付）
   -> 通知用户支付

4. [决策] 用户是否在规定时间(30分钟)内完成支付？
   - 否 -> 释放库存 -> 订单状态变更为"已取消" -> 终点
   - 是 -> 进入步骤5

5. 支付确认
   -> 订单状态变更为"已支付"
   -> 发送通知给仓库系统

6. 仓库发货
   -> 订单状态变更为"已发货"
   -> 发送物流信息给用户

7. [决策] 用户是否确认收货？
   - 否(超7天) -> 自动确认收货 -> 进入步骤8
   - 是 -> 进入步骤8

8. 订单完成
   -> 订单状态变更为"已完成"
   -> 释放资金给商家（如有担保交易）
   -> 终点

异常路径：
- 支付超时: 30分钟未支付 -> 释放库存 -> 订单取消
- 发货失败: 通知客服人工处理
- 退款申请: 在"已支付"或"已发货"状态下用户可申请退款 -> 进入退款子流程
```

## 示例 6: 信息未找到

### 输入 (Caller)

```
file_path: /path/to/api-doc.pdf
goal: 提取所有涉及"限流(rate limiting)"的配置
```

### 正确输出

```
未找到限流配置：本文档中不包含限流(rate limiting)相关的配置说明。文档实际涵盖的内容为：认证配置、日志配置、数据库连接配置、缓存配置。如需限流配置，建议查阅网关层文档或运维配置手册。

```

## 输出格式速查

| 场景 | 输出开头 | 典型长度 |
|------|----------|----------|
| 原文提取(章节/段落) | 直接开始内容（保留原标题层级） | 长 |
| 数据提取(表格/数值) | Markdown 表格或列表 | 中 |
| 结构化描述(架构/流程/UI) | 按框架逐段输出 | 长 |
| 未找到信息 | "未找到{信息}：{原因}" | 短(1-2句) |
| 文件无法读取 | "无法读取文件 {path}：{原因}" | 短(1句) |

**绝对禁止的输出模式：**
- "好的/明白了/让我来..." (preamble)
- "以上就是/如果还需要/希望对你有帮助" (postamble)
- "我建议/或许可以/建议查一下" (未请求的建议)
- "经过分析，我发现/从图中可以看出" (元描述)
