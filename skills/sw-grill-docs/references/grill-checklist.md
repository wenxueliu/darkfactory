# GrillChecklist: 文档对照质询检查清单

完整的 4 阶段质询流程，含 pass/fail 标准和每阶段的具体质询程序。

---

## 阶段 0: 质询准备

在开始质询前，完成以下准备：

```
□ 目标文档已加载（设计文档或计划文档的完整内容）
□ CONTEXT.md 已读取（如果存在）
□ CONTEXT-MAP.md 已读取（如果存在）
□ docs/adr/ 目录已扫描，ADR 编号列表已建立
□ config.yaml 已读取（business_domain, communication_language）
□ 质询深度已确定（Quick / Standard / Deep）
```

### 质询深度选择矩阵

| 深度 | 文件数 | 新概念数 | 架构影响 | 何时使用 |
|------|--------|---------|---------|---------|
| **Quick** | 1-3 | 0-2 | 无 | 小改动、简单功能 |
| **Standard** | 3-10 | 2-5 | 跨模块 | 中等特性、新模块 |
| **Deep** | 10+ | 5+ | 多服务 | 架构设计、新服务 |

## 阶段 1: 术语审计 (Glossary Audit)

### 快速检查（Quick）

扫描目标文档，提取所有可能是领域术语的名词/动词：

```
对每个候选术语:
  1. 在 CONTEXT.md 中搜索该术语
  2. 如果找到 → 检查用法是否一致
  3. 如果未找到 → 判断是否是新概念
```

**Quick 检查项（10 项快速扫描）:**
1. 文档中使用了哪些名词作为核心概念？
2. 这些概念在 CONTEXT.md 中有定义吗？
3. 如果有定义，用法一致吗？
4. 如果有术语在 CONTEXT.md 中但是含义不同 → CHALLENGE
5. 如果引入了明显的新概念 → 标记为 NEW
6. 如果概念与已有术语相近但用词不同 → 提议使用已有术语
7. 文档中的术语之间关系是否清晰？（"A 属于 B？A 产生 B？A 依赖 B？"）
8. 是否有口语化或模糊的表达需要精确化？
9. 是否有同义词混用（"A 和 B 指的是同一个东西"）？
10. 是否有歧义词（"X 在不同上下文中含义不同"）？

### 标准检查（Standard）

在 Quick 基础上增加：

```
对 CONTEXT.md 中的每个已定义术语:
  1. 在目标文档中搜索该术语
  2. 如果文档涉及该术语的领域但未使用 → QUESTION: "你的设计涉及 X 领域，但未使用术语 Y。这是故意的简化还是遗漏？"
  3. 如果文档使用了该术语但含义偏离 → CHALLENGE
```

**Standard 增加检查项:**
11. CONTEXT.md 中的术语在你的文档中是否被正确使用？
12. 你的文档是否涉及了 CONTEXT.md 定义的领域但未使用其术语？
13. 如果有示例对话，你的文档是否与示例中展示的术语交互模式一致？
14. 新引入的术语是否与已有术语形成自然的层次/关系？

### 深度检查（Deep）

在 Standard 基础上增加：

```
构建术语关系图:
  1. 列出目标文档中所有核心概念
  2. 列出 CONTEXT.md 中所有相关概念
  3. 画出关系（ownership, dependency, event flow, identity）
  4. 检查关系是否一致、完整、无循环依赖
```

**Deep 增加检查项:**
15. 新术语与已有术语之间的关系是否完全定义？
16. 术语关系是否存在冲突（A→B 在 CONTEXT.md, B→A 在你的文档中）？
17. 是否存在"隐式术语"——被使用但从未被命名的概念？

### pass/fail 标准 (Phase 1)

| 条件 | 判定 |
|------|------|
| 所有核心术语在 CONTEXT.md 中有定义且用法一致 | **PASS** |
| 有术语需要用户澄清但不是矛盾 | **CONCERNS** — 继续质询 |
| 文档使用术语的方式与 CONTEXT.md 定义直接矛盾 | **CONFLICT** — 阻止继续 |

## 阶段 2: ADR 合规检查 (ADR Compliance Check)

### 快速检查（Quick）

```
对目标文档中的每个技术/架构决策:
  1. 扫描 docs/adr/ 目录中的所有 ADR 标题和摘要
  2. 判断是否有 ADR 与此决策相关
  3. 如果存在相关 ADR → 检查一致性
  4. 如果无相关 ADR 且决策不需要 ADR → PASS
```

**Quick 检查项:**
1. 目标文档中有多少显式的技术/架构决策？
2. 其中多少与已有 ADR 相关？
3. 相关 ADR 的决策与目标文档的决策是否一致？
4. 是否有决策明显与已有 ADR 矛盾？
5. 是否有决策是新的重大架构选择？

### 标准检查（Standard）

在 Quick 基础上增加：

```
对每个相关 ADR:
  1. 重新读取 ADR 全文（不仅是摘要）
  2. 检查 "Consequences" 部分（如果存在）——目标文档是否考虑了这些后果？
  3. 检查 "Considered Options" 部分——目标文档选择的方案是否曾被 ADR 考虑并否决？
```

**Standard 增加检查项:**
6. 目标文档是否考虑了相关 ADR 的后果（Consequences）？
7. 目标文档的选择是否恰好是某个 ADR 中考虑并否决的方案？
8. 目标文档的选择是否让相关 ADR 的部分内容过时（需要 superseded）？

### 深度检查（Deep）

在 Standard 基础上增加：

```
对目标文档的完整技术栈和架构:
  1. 评估 ADR 覆盖范围——哪些决策已有 ADR，哪些没有
  2. 对于没有 ADR 覆盖的新重大决策：
     a. 评估是否满足 3 个 ADR 创建条件
     b. 准备 ADR 创建提议
  3. 对于隐式否决的 ADR：
     a. 确认否决是否是故意的
     b. 准备 supersede ADR 提议
```

**Deep 增加检查项:**
9. 整体 ADR 覆盖度评估——多少架构决策有 ADR 支撑？
10. 目标文档如果有架构级决策，ADR 是否需要更新或新建？
11. 多个相关 ADR 之间是否有逻辑链？文档是否尊重了这个逻辑链？

### ADR 创建条件检查

对每个候选 ADR，在提议创建前确认**全部 3 个条件**：

| 条件 | 检查 |
|------|------|
| **难以逆转** | 以后改变这个决定的代价是否显著？（不是"改一行代码就行"） |
| **缺少上下文会令人困惑** | 未来的读者看到实现时是否会困惑"为什么这么做"？ |
| **真实权衡的结果** | 是否存在真实的其他选择，你为特定原因选择了一个？ |

**任一条件不满足 → 跳过 ADR。**

### pass/fail 标准 (Phase 2)

| 条件 | 判定 |
|------|------|
| 所有决策与相关 ADR 一致 | **PASS** |
| 存在与 ADR 不一致但需要用户确认的决策 | **CONCERNS** — 继续质询 |
| 存在与 ADR 直接矛盾且未意识到的决策 | **CONFLICT** — 阻止继续 |

## 阶段 3: 场景压力测试 (Scenario Stress-Test)

### 快速检查（Quick）

```
对目标文档中的每个关键领域关系:
  1. 找出核心流程（主路径）
  2. 为每个流程找一个边缘情况
  3. 检查文档是否有对应处理
```

**Quick 检查项:**
1. 核心流程是否在文档中有完整描述？
2. 核心流程中的每个步骤是否有明确的行为定义？
3. 主要的错误路径是否被覆盖？
4. 是否至少有一个边缘场景被明确讨论？

### 标准检查（Standard）

在 Quick 基础上增加：

```
对每个关键领域关系:
  1. 正面场景: 正常流程能否走通？
  2. 边缘场景: 边界条件是否定义？
  3. 冲突场景: 两个概念交互的模糊地带是否有明确行为？
```

**Standard 增加检查项:**
5. 跨模块边界的行为是否定义清楚？
6. 并发/竞争条件下的行为是否被考虑？
7. 数据一致性边界是否明确（强一致 vs 最终一致）？
8. 失败模式是否被讨论（依赖不可用时会发生什么）？

### 深度检查（Deep）

在 Standard 基础上增加：

```
1. 构造"不可能但发生了"的场景——系统是否优雅降级？
2. 状态机分析——所有状态转换是否被覆盖？
3. 时序分析——事件顺序是否有隐含假设？
4. 规模分析——设计在极端规模下是否仍然成立？
```

**Deep 增加检查项:**
9. 所有可能的状态转换是否被文档覆盖？
10. 极端情况下的系统行为是否有定义（不只是"ERROR"）？
11. 是否存在隐含的时序依赖（"A 必须在 B 之前"但未明确声明）？

### pass/fail 标准 (Phase 3)

| 条件 | 判定 |
|------|------|
| 所有关键场景有明确的行为定义 | **PASS** |
| 存在未覆盖的边缘情况但可以被实现者自行处理 | **PASS** — 标记为 NOTE |
| 存在设计空白——核心场景的行为完全未定义 | **GAP** — 需要补充 |
| 存在两个场景的行为描述互相矛盾 | **CONFLICT** — 阻止继续 |

## 阶段 4: 代码交叉验证 (Code Cross-Reference)

### 仅在以下条件触发

此阶段仅在目标文档**对现有系统行为做了声称**时执行。如果目标文档是纯绿地设计（无现有代码），跳过此阶段并标记 "N/A"。

### Quick 检查

```
对目标文档中关于现有系统的每个声称:
  1. 找到声称涉及的代码路径
  2. 验证代码行为是否与声称一致
  3. 不一致 → CHALLENGE
```

### Standard 检查

在 Quick 基础上增加：

```
1. 不仅验证"做了什么"，还验证"怎么做"的声称
2. 检查声称的性能特征是否有证据支持
3. 检查声称的边界/限制是否与实际代码一致
```

### Deep 检查

在 Standard 基础上增加：

```
1. 验证声称的"为什么不"——如果文档说"因为 X 限制所以我们不做 Y"，验证 X 确实存在
2. 验证声称的依赖关系——如果文档说"A 依赖 B"，检查代码中 A 确实引用了 B
3. 检查代码演化趋势——最近提交是否显示了与文档声称相反的方向？
```

### pass/fail 标准 (Phase 4)

| 条件 | 判定 |
|------|------|
| 所有声称与代码行为一致 | **PASS** |
| 存在轻微不精确但意图正确 | **PASS** — 标记为 NOTE |
| 声称的行为与代码行为明显不一致 | **CONFLICT** — 阻止继续 |

## 质询执行顺序

推荐的质询步骤（按重要性降序）：

```
第 1 步: 术语审计 → 最快发现概念层面的问题
第 2 步: ADR 合规检查 → 最容易发现架构层面的矛盾
第 3 步: 场景压力测试 → 暴露设计空白和边界模糊
第 4 步: 代码交叉验证 → 仅在需要时执行，用证据说话
```

## 文档更新协议

### 更新 CONTEXT.md

```
当术语被澄清时:
  1. 如果术语已存在但定义需要更新 → 立即 Edit CONTEXT.md
  2. 如果术语是新的 → 按 CONTEXT-FORMAT.md 格式立即添加
  3. 如果发现歧义 → 在 "Flagged ambiguities" 中记录
```

**规则:** 不批量处理。每个术语决定后立即写入。

### 创建 ADR

```
当一个新的重大决策被做出时:
  1. 检查 3 个创建条件（难以逆转 + 令人困惑 + 真实权衡）
  2. 如果全部满足 → 按 ADR-FORMAT.md 格式创建
  3. 编号 = 扫描 docs/adr/ 中最高编号 + 1
  4. 如果 docs/adr/ 不存在 → 创建目录
```

## Verification

质询完成前确认:
- [ ] 所有 CHALLENGE 都包含了具体的引用（CONTEXT.md 行或 ADR 编号）
- [ ] 所有 CONFLICT 都被标记为阻断性问题
- [ ] CONTEXT.md 更新已写入（如果有）
- [ ] 新的 ADR 已创建（如果有）
- [ ] 质询报告已按标准格式输出
