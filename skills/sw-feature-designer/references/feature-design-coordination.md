# 特性设计协调 (Feature Design Coordination)

## 核心理念

特性设计的目标是产出一份**跨服务的 "大图"**——让 per-service 设计者可以独立工作，让 E2E 设计者有明确的用户旅程可依据。你不是在设计服务内部，而是在定义服务之间的边界和契约。

**特性设计文档是 Stage 2 (per-service 设计) 和 Stage 3 (E2E 设计) 的输入契约。**

## 协调流程 (4 步)

### 第 1 步: 知识库优先 (Knowledge Base First)

**目标:** 在开始设计之前，先理解已有的约束和可复用的模式。

| 查询目标 | 在知识库中的位置 | 要回答的问题 |
|---------|----------------|-------------|
| 架构决策 | `knowledge-base/decisions/ADR-*.md` | 哪些已有 ADR 会约束本次设计？ |
| 可复用模式 | `knowledge-base/patterns/` | 有没有已解决过类似问题的模式？ |
| 经验教训 | `knowledge-base/lessons/` | 过去类似场景踩过什么坑？ |
| 服务注册表 | `service-registry.yaml` | 哪些服务存在？它们的 API 和依赖是什么？ |
| 需求规格 | `requirements/{id}.md` | 需求的具体约束和 AC 是什么？ |
| 头脑风暴输出 | `designs/{id}-brainstorm.md` | 推荐的技术方向是什么？ |

**查询后行动:**
1. 给人类一个简短摘要: "根据知识库，已有 {N} 个相关 ADR。本次设计受 {M} 个已有决策约束。涉及 {X} 个服务。"
2. 标记冲突: 新需求与已有 ADR 矛盾 → 立即标记，不默默绕过
3. 标记缺口: 知识库缺乏相关信息 → 记录，设计完成后回填

### 第 2 步: 识别受影响服务 (Service Impact Mapping)

**目标:** 从需求规格中识别哪些服务需要变更。

**识别方法:**

1. **从需求 AC 反推:** 每个 AC 涉及什么操作？这些操作在哪个服务？
2. **从用户旅程反推:** 旅程的每个步骤调用哪些服务？
3. **从数据依赖反推:** 需求涉及的数据在哪个服务的数据库中？

**输出:** 填充模板 Section 2 "服务影响分析" 表:
- 每个受影响服务一行
- 标注影响类型 (新增 API / 修改 API / 新增 consumer / 新增事件 / 数据迁移 / UI 变更)
- 标注跨服务依赖
- 评估风险等级

**微服务模式:** 从 `service-registry.yaml` 获取服务列表，交叉比对。
**单体模式:** 识别受影响的模块/包，输出到 Section 2。

### 第 3 步: 渐进式填充 (Progressive Fill)

**目标:** 从粗到细，逐层展开特性设计。不是一次写完 9 个章节。

#### 轮次 1: 骨架 (30%)
- Section 1: 特性总览 + 成功标准
- Section 2: 服务影响分析
- Section 5: 服务交互设计 (调用序列 + SLA + 降级)

**检查点:** "服务影响范围是否合理？服务交互是否有循环依赖？"

#### 轮次 2: 用户视角 (60%)
- Section 3: 用户旅程设计 (交互流程 + 关键交互时刻 + 交互状态矩阵)
- Section 4: 页面设计 (如果涉及 UI)

**检查点:** "用户旅程是否覆盖了所有 AC？交互状态矩阵是否覆盖了每步的所有状态？"

#### 轮次 3: 契约与部署 (100%)
- Section 6: 跨服务契约 (contract-first: 契约文件先于实现)
- Section 7: 部署策略 (发布序列 + 特性开关 + 回滚 + 监控)
- Section 8: 开放问题
- Section 9: 下游引用

**检查点:** "每个跨服务调用是否有明确的契约？部署序列是否考虑了依赖顺序？"

### 第 4 步: 输出与过渡

**输出产物:**
- 写入 `designs/{requirement_id}-design.md`

**过渡条件 (可以进入 Stage 2 的条件):**
- [ ] 9 个章节完整 (标注 N/A 的除外)
- [ ] 服务影响分析覆盖了所有受影响服务
- [ ] 每个用户旅程覆盖完整 happy path，标注了 AC 编号和涉及服务
- [ ] 每个跨服务调用（如适用）有 SLA 和降级策略
- [ ] 每个跨服务契约（如适用）已定义提供方和消费方
- [ ] 部署策略包含发布序列和回滚方案
- [ ] 开放问题已记录

**完成确认语:** "特性设计文档完成。{N} 个服务受影响。用户旅程覆盖 {M} 个 AC。{X} 个跨服务契约已定义。准备好进入 Stage 2 (per-service 详细设计) 吗？"

## 与其他阶段的集成

| 上游 | 下游 | 集成方式 |
|------|------|---------|
| 需求规格 | 特性设计 | 需求 AC → 用户旅程步骤 |
| 头脑风暴 | 特性设计 | 推荐方向 → 技术决策种子 |
| 特性设计 | Per-service 设计 (Stage 2) | 服务影响分析 → sw-service-designer 输入 |
| 特性设计 | E2E 设计 (Stage 3) | 用户旅程 + 服务交互 → sw-e2e-designer 输入 |
