# 高精度模式 (High Accuracy Mode)

Phase 3: Momus（莫摩斯，希腊讽刺之神）审查循环。当用户选择 "High Accuracy Review" 时激活。这是一个不可协商的承诺——计划必须通过严格的审查循环，直到获得 OKAY 裁决。

---

## 触发条件

用户在计划呈现后的选择提示中选择 "High Accuracy Review"。

```
用户: "High Accuracy Review"
→ 立即进入 Momus 审查循环
```

---

## Momus 审查循环协议（绝对要求）

### 完整循环流程

```
1. 提交计划给 hw-plan-reviewer (Momus)
2. 等待审查结果
3. 如果裁决是 OKAY → 退出循环，进入交接阶段
4. 如果裁决是 REJECT → 必须修复所有问题并重新提交
5. 重复直到 OKAY
```

### 调用方式

```
委托给 hw-plan-reviewer:
{plan-path}  # 仅文件路径字符串作为输入
```

**关键**: 提交给 Momus 时，仅提供计划文件路径。不包裹解释、markdown 或对话性文字。

**注意**: hw-plan-reviewer 可能会附加上下文（如系统指令），但这由平台运行时处理并由 Momus 预期——你只需要提供文件路径。

---

## 高精度模式的关键规则

### 1. 不找借口 (NO EXCUSES)

如果 Momus 拒绝，你必须修复它。无可争辩。

- "This is good enough" → **不可接受**
- "The user can figure it out" → **不可接受**
- "These issues are minor" → **不可接受**
- "I already addressed this in the interview" → **不可接受 — 如果它不在计划中，它就不存在**

### 2. 修复所有问题 (FIX EVERY ISSUE)

处理 Momus 的**所有**反馈，而非仅部分。

- Momus 提出 5 个问题 → 修复所有 5 个
- 部分修复 → Momus 将再次拒绝

### 3. 持续循环 (KEEP LOOPING)

没有最大重试限制。

- 第 1 次拒绝 → 修复并重新提交
- 第 2 次拒绝 → 修复并重新提交
- 第 10 次拒绝 → 修复并重新提交
- 循环直到 "OKAY" 或用户明确取消

### 4. 质量是不可协商的 (QUALITY IS NON-NEGOTIABLE)

用户要求了高精度。

- 他们信任你提供一个防弹计划
- Momus 是守门人
- 你的工作是让 Momus 满意，而非与它争论

### 5. 绝不与审查者争论

- Momus 的推理可能有缺陷，但它的检查是客观的（文件是否存在，任务是否可执行）
- 如果 Momus 错误地标记了某些东西 → 在计划中澄清，而非在响应中争论
- 目标: 创建一个没有审查者可以发现问题的计划

---

## Momus 审查什么（4 项检查）

回顾 hw-plan-reviewer 的检查范围:

1. **引用验证 (Reference Verification)** — 在计划中引用的文件是否存在？它们包含声称的内容吗？
2. **可执行性 (Executability)** — 开发者能否开始工作？每个任务是否有明确的起点？
3. **关键阻塞 (Critical Blockers)** — 缺失信息使工作完全停滞？计划各部分之间的矛盾？
4. **QA 场景可执行性 (QA Scenario Executability)** — 每个任务是否有具体的工具 + 步骤 + 预期结果？

---

## "OKAY" 意味着什么

Momus 仅在以下条件满足时批准:

- 100% 的文件引用已验证
- 零严重失败的文件验证
- >=80% 的任务具有清晰的引用来源
- >=90% 的任务具有具体的验收标准/QA 场景
- 零任务需要对商业逻辑的假设
- 清晰的大图和流程理解
- 零关键红旗

**在看到 Momus 的 "OKAY" 之前，计划尚未准备好。**

## 处理 REJECT

### 收到 REJECT 时

```
[Momus 返回 REJECT]
Summary: {1-2 句话总结问题}
Blocking Issues:
1. {具体问题 + 需要怎么改}
2. {具体问题 + 需要怎么改}
3. {具体问题 + 需要怎么改}
```

### 修复优先级

1. **文件引用问题（最高优先级）** — 立即搜索正确路径并更新计划
2. **矛盾问题** — 重新审查计划逻辑，消除不一致之处
3. **遗漏信息** — 补充具体文件位置、做什么、明确的验收标准
4. **QA 场景不足** — 添加具体工具 + 步骤 + 断言

### 修复后

1. 静默更新计划文件
2. 如果合适，更新草稿（如果草稿未删除）
3. 重新提交给 Momus
4. 告知用户: "已处理 Momus 的反馈，重新提交审查中..."

---

## 循环终止条件

### 正常终止

```
[Momus 返回 OKAY]
→ 退出循环
→ 进入交接阶段 (references/handoff-protocol.md)
```

### 用户取消

```
用户: "不要再审查了，现在就开始执行"
→ 退出循环
→ 记录: 计划在 N 次审查迭代后被用户接受但未通过 Momus
→ 进入交接阶段
```

### 用户请求变更范围

```
用户: "等等，我还需要加上 X 功能"
→ 退出循环
→ 回到访谈模式（重新打开清关清单）
→ 更新范围，生成新的计划
→ 高精度模式需要重新请求
```

---

## 高精度模式反模式

### NEVER

- 跳过 Momus 审查（用户选择了高精度）
- 第一次拒绝后放弃
- 只修复部分阻塞问题
- 未经修改直接重新提交给 Momus
- 与 Momus 争论裁决
- 在 OKAY 之前向用户呈现最终计划

### ALWAYS

- 修复所有阻塞问题
- 静默更新计划
- 持续循环直到 OKAY
- 告知用户循环进度
- 仅在 OKAY 后进入交接
