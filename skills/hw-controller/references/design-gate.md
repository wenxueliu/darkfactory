# 设计门禁检查

## 触发时机

设计文档完成，进入任务拆分之前。

## 门禁清单

### G1: 完整性检查

- [ ] 所有必填章节完整（架构设计、API/接口设计、错误处理、安全设计、测试策略）
- [ ] 每个技术决策有被拒绝的替代方案和拒绝理由
- [ ] API 端点定义了请求/响应 schema 和错误码
- [ ] 状态机覆盖了所有合法转换和非法转换的防护

### G2: 可实施性检查

- [ ] 每个组件有明确的输入/输出契约
- [ ] 错误处理覆盖了至少 5 种异常场景
- [ ] 部署注意事项已填写（迁移、特性开关、回滚方案至少各一项）
- [ ] 测试策略每层有预估用例数

### G3: 安全就绪检查

- [ ] 安全设计表至少覆盖：认证、授权、输入校验、数据保护
- [ ] 每个 API 端点标注了授权要求
- [ ] 敏感数据（PII、凭证、支付）的处理方式已明确

### G4: 知识沉淀检查

- [ ] 至少 1 个 ADR 写入知识库（如果设计包含重要技术决策）
- [ ] 设计文档中有设计决策表（≥ 2 个决策）
- [ ] 开放问题已记录（不能以 "TODO" 或 "待定" 结尾而无负责人）

## 门禁输出

```
PASS: → 进入任务拆分阶段
FAIL: → 标记阻塞原因，回到设计修订。最大重试 3 轮。
       3 轮后仍未 PASS → 升级到人工决策。
```

## 输出文件

门禁结果写入 `{project-root}/_bmad/memory/hw-shared/designs/{requirement_id}-design-gate.md`
