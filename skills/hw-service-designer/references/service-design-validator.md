# 服务设计验证器 (Service Design Validator)

## 触发时机

Per-service 设计文档 (`designs/{requirement_id}-service-{service_id}-design.md`) 完成后，进入 Stage 3 之前。

## 验证清单

### V1: 完整性

- [ ] S1-S8 章节完整 (按服务类型模板检查对应章节)
- [ ] S2: 组件图 + 数据流 + 组件职责表完整
- [ ] S3: 所有端点有请求/响应 schema + 错误码
- [ ] S4: 状态机覆盖所有合法转换 + 至少 1 条非法转换防护
- [ ] S5: ≥ 5 种异常场景 (后端) / ≥ 5 种错误场景 (前端/BFF)
- [ ] S6: 安全设计覆盖认证/授权/输入校验/数据保护

### V2: UT 设计质量

- [ ] 每个组件 ≥ 2 UT 用例 (1 happy + 1 error/boundary)
- [ ] 每个组件 ≥ 1 edge 用例
- [ ] 每条 UT 用例附带数据构造代码块 (输入 + mock + 预期输出)
- [ ] UT 用例的输入/输出为具体值，无 `{占位符}`
- [ ] UT 用例 → 需求 AC 追溯完整

### V3: API 测试设计质量

- [ ] 每个端点 ≥ 3 API 用例 (正常 ×1 + 异常 ×1 + 认证/权限 ×1)
- [ ] API JSON 文件已生成且有效 JSON
- [ ] API JSON 中的 `item[].name` 与设计文档的用例 ID 一一对应
- [ ] API 用例的请求体为具体值，无 `{占位符}`

### V4: 可执行性

- [ ] 组件职责表足够清晰，TDD Agent 能从中确定测试范围
- [ ] UT 数据构造代码块可直接复制到测试代码中
- [ ] API JSON 文件可直接用 Newman 执行
- [ ] 安全设计方案有对应的验证方式

## 输出

```
PASS: → 该服务设计完成，等待其他服务设计完成后进入 Stage 3
FAIL: → 标记缺失项，回到服务设计修订。最大重试 3 轮。
       3 轮后仍未 PASS → 升级到人工决策。
```
