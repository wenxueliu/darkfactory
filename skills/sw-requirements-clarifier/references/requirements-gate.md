# 需求门禁检查

## 触发时机

需求规格文档完成，进入设计阶段之前。

## 门禁清单

每一项必须 PASS 才能进入设计阶段。FAIL 项需标记阻塞原因并回到需求澄清。

### G1: 完整性检查

- [ ] 所有必填章节完整（问题陈述、范围、用户故事、验收标准）
- [ ] 每个用户故事有 ≥ 2 个验收标准（正常路径 + 边界/异常）
- [ ] 非功能需求已定义（性能、安全、可用性至少各一项）
- [ ] 依赖项已识别并标注状态

### G2: 可测量性检查

- [ ] 所有验收标准可被客观验证（不依赖主观判断）
- [ ] 非功能需求有具体的度量阈值（不是 "fast enough" 而是 "< 200ms"）
- [ ] "完成"的定义不会引起歧义

### G3: 价值对齐检查

- [ ] 价值评估已完成（用户价值、业务价值、战略对齐）
- [ ] 优先级与战略目标一致
- [ ] 如果评分为 P3（不做），此需求应被归档而非进入设计

### G4: 风险就绪检查

- [ ] 所有高风险项（概率 H × 影响 H）有缓解措施
- [ ] 假设已被记录（至少识别 3 个关键假设）
- [ ] 如果 Premise 1（AI 代码质量假设）相关，已标注验证状态

## 门禁输出

```
PASS: → 进入设计阶段
FAIL: → 标记阻塞原因，回到需求澄清。最大重试 3 轮。
       3 轮后仍未 PASS → 升级到人工决策。
```

## 输出文件

门禁结果写入 `{project-root}/_context/memory/sw-shared/requirements/{requirement_id}-gate.md`
