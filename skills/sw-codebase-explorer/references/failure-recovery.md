# failure-recovery.md — 搜索失败恢复与停止条件

## 何时加载

当搜索结果不理想时加载本文档 — 包括未找到结果、结果太少、结果过多、结果不相关等情况。

---

## 1. 搜索结果分类与应对策略

### 1.1 无结果 — 什么都没找到

**先自我检查（不增加搜索轮次）：**
1. 搜索词是否有拼写错误？
2. 搜索范围是否太窄（只搜了特定目录、特定文件类型）？
3. 是否使用了不正确的工具（如用 LSP 搜注释内容）？
4. 大小写敏感设置是否正确？

**第一轮恢复策略（并行执行以下 3 项）：**
1. **放宽关键词** — 使用更通用的术语。例如 `authenticateUser` → `auth`，`PaymentGateway` → `payment`
2. **扩大搜索范围** — 从特定目录扩展到整个项目，从特定文件类型扩展到所有文件
3. **切换工具类型** — 如果 grep 无结果，尝试 glob 按文件名找；如果 LSP 无符号，尝试 grep 文本搜索

**第二轮恢复策略（如果第一轮仍无结果）：**
1. **检查命名约定** — 项目可能使用不同命名风格（camelCase vs snake_case、缩写 vs 全拼）。尝试变体
2. **搜索注释和文档** — 可能功能在 README/CHANGELOG/注释中有提及但尚未实现
3. **git log 搜索** — 功能可能在历史提交中存在过，后来被删除或重命名

```
搜索失败诊断流程：

无结果
  ├─ 拼写检查 → 修正 → 重搜
  ├─ 扩大范围 → 全项目 → 重搜
  ├─ 切换工具 → grep → glob → LSP → ast_grep → 依次并行
  ├─ 搜索注释/文档 → 功能可能未实现
  └─ git log --grep → 历史中是否存在过
```

### 1.2 结果过少 — 只找到 1-2 个匹配

**常见原因与应对：**

| 原因 | 应对 |
|------|------|
| 搜索词太具体 | 使用更宽泛的模式，去掉限定词 |
| 只用了 1 种工具 | 并行使用另外 2 种工具交叉验证 |
| 文件类型过滤太严格 | 扩展文件类型范围 |
| 功能确实只有少量实现 | 确认后如实报告，标注"已穷尽搜索" |

**操作：** 如果找到 1-2 个结果，它们是一个好的起点。对每个结果：
- 用 `lsp_find_references` 查该文件中的关键符号被哪些文件引用
- 用 `lsp_symbols` 了解该文件结构，发现相关符号
- 用 grep 搜索该文件中出现的独特术语

### 1.3 结果过多 — 超过 50 个匹配

**分层收敛策略：**

```
第一层：分类统计
  - 按目录分组统计匹配数
  - 识别结果模式（例如：80% 在 tests/ 中，15% 在 src/ 中）

第二层：精准过滤
  - 限制到核心目录（src/、lib/、app/ 等）
  - 排除测试文件（除非搜索目标是测试）
  - 排除 vendor/node_modules 等第三方代码
  - 添加更具体的搜索词

第三层：排序与优先级
  - 按修改时间排序，优先展示最近修改的文件
  - 按路径深度排序，核心模块优先于工具函数
  - 列出最相关的 10 个文件，其余归类总结

报告格式：
  核心文件（最相关 10 个）: ...
  其余约 N 个匹配分布在 [目录A]、[目录B]...
```

### 1.4 结果不相关 — 找到了东西但不对

**诊断问题：**

1. **意图理解错误** — 你搜索的和你应该搜索的不一样。回到 `<analysis>` 重新思考调用者的实际需求
2. **关键词歧义** — 搜索词在代码库中有多重含义。例如 `queue` 可能是消息队列，也可能是数据结构
3. **上下文缺失** — 你需要更多上下文才能正确搜索。考虑向调用者确认

**应对：**
- 如果可能是意图理解错误 → 输出你的当前理解，请求调用者确认或提供更多线索
- 如果可能是关键词歧义 → 添加排除词缩小范围，或添加限定词明确上下文
- 如果可能需要更多上下文 → 诚实说明，不要猜测

## 2. 搜索范围调节

### 2.1 何时扩大搜索

- 在特定目录搜索无结果 → 扩展到父目录或整个项目
- 搜索特定文件类型无结果 → 扩展到所有文件类型
- 使用精确匹配无结果 → 切换到模糊匹配、子串匹配
- 搜索源码无结果 → 扩展到配置文件、文档、注释

### 2.2 何时缩小搜索

- 结果超过 50 个且大部分不相关 → 添加目录限制、文件类型限制
- 搜索词太通用（如 `handle`、`process`）→ 添加更具体的上下文词
- 第三方依赖的匹配过多 → 排除 `node_modules/`、`vendor/`、`site-packages/`、`.git/`

### 2.3 库代码 vs 项目代码

**重要：** 第三方库代码不是调用者关心的内容。默认排除：
- `node_modules/`
- `vendor/`
- `site-packages/`
- `.git/`
- `dist/`、`build/`、`target/`（构建产物）
- `*.min.js`、`*.bundle.js`（压缩文件）

除非调用者明确要求搜索第三方代码。

## 3. 停止条件

### 3.1 必须停止的情况

| 条件 | 说明 |
|------|------|
| 已完成 3 轮搜索 | 每轮包含 3+ 并行工具调用。3 轮后如实报告 |
| 通过 2+ 种工具交叉验证确认结果 | 已穷尽搜索，信心高 |
| 调用者提供的线索已全部用完 | 无更多搜索方向可尝试 |

### 3.2 不应停止的情况

| 条件 | 说明 |
|------|------|
| 只用了 1 种工具 | 继续用第二种、第三种工具交叉验证 |
| 只找到 1 个匹配就下结论 | 可能还有更多相关位置 |
| 搜索结果中有疑问点 | 追问调用者或进一步探索疑问点 |
| 路径是相对路径 | 必须转换为绝对路径后才算完成 |

### 3.3 搜索未果时的最终报告

当确认（通过 3 种以上工具、3 轮搜索后）某物不存在时，按以下格式报告：

```
<analysis>
**Literal Request**: [字面请求]
**Actual Need**: [实际需求]
**Success Looks Like**: [理想结果]
</analysis>

<results>
<files>
（经穷尽搜索，未找到匹配的文件）
</files>

<answer>
未在代码库中找到 [搜索目标]。

已执行的搜索（3 轮，共使用 N 种工具）：
- 第 1 轮：grep "[pattern1]" + glob "**/target*" + LSP symbols in [scope]
- 第 2 轮：grep "[pattern2]" (放宽) + ast_grep_search "[structure]" + git log --grep="[keyword]"
- 第 3 轮：grep "[pattern3]" (注释/文档) + glob "[wider-pattern]" + LSP find_references on [related symbol]

排除的假设：
1. [假设A] — 已通过 [方法] 排除
2. [假设B] — 已通过 [方法] 排除

结论：[功能/代码] 在当前代码库中不存在。可能的原因：
- 尚未实现（最可能）
- 使用了不同的命名约定
- 位于独立仓库/子模块中
</answer>

<next_steps>
- 如需继续搜索：请提供更多线索（可能的相关文件名、类名、提交者、提交时间）
- 如需实现该功能：这是新增功能的切入点，可以从 [建议的文件路径] 开始
</next_steps>
</results>
```

## 4. 歧义处理

### 4.1 多义搜索词

当搜索词有多种可能含义时，先列出可能含义，再分别搜索：

```
搜索词 "queue" 的可能含义：
1. 消息队列（SQS, RabbitMQ, Kafka）
2. 数据结构（FIFO queue, PriorityQueue）
3. 任务队列（Bull, Celery）
4. 队列服务（queue worker, consumer）

策略：在 <analysis> 中列出含义，根据调用者上下文推断最可能的含义。
如果歧义过大，简要说明歧义，并请求调用者明确。
```

### 4.2 命名的多种可能

当搜索一个概念但不知道项目中的具体命名时，尝试以下变体：

| 概念 | 可能的命名 |
|------|-----------|
| 用户认证 | `auth`, `authenticate`, `login`, `signin`, `session`, `credential`, `token`, `jwt`, `oauth` |
| 支付 | `pay`, `payment`, `charge`, `billing`, `invoice`, `transaction`, `checkout` |
| 配置 | `config`, `settings`, `options`, `preferences`, `.env`, `.rc`, `.yaml`, `.toml` |
| 日志 | `log`, `logger`, `logging`, `audit`, `trace`, `telemetry` |
| 缓存 | `cache`, `redis`, `memcache`, `store`, `temp`, `session` |
| 数据库 | `db`, `database`, `repository`, `dao`, `model`, `schema`, `migration`, `seed` |

**策略：** 第一轮搜索使用最可能的 3-4 个变体，并行执行。

## 5. 快速决策表

| 症状 | 诊断问题 | 立即行动 |
|------|----------|----------|
| 无结果 | 搜索词太精确？ | 放宽关键词 + 扩大范围 + 切换工具 |
| 只有 1-2 个结果 | 只用了 1 种工具？ | 对找到的符号查引用 + 交叉验证 |
| 超过 50 个结果 | 搜索词太宽泛？ | 添加限定词 + 限制目录 + 排除测试/第三方 |
| 结果不相关 | 关键词歧义？ | 在 analysis 中列出歧义，选择最佳含义重搜 |
| 路径是相对的 | 没有转换为绝对路径？ | 立即补全为绝对路径 |
| 调用者会追问 | 回答不够完整？ | 检查是否回答了实际需求，补充 <answer> 和 <next_steps> |
| 3 轮后仍无结果 | 功能不存在？ | 按搜索未果最终报告格式如实汇报 |
