# 架构分析词汇表 (Architecture Vocabulary)

分析代码库架构时使用的精确词汇表。来自 `improve-codebase-architecture` 技能的核心模式。一致的语言是精确分析的基础——不要使用 "组件"、"服务"、"API"、"边界" 等模糊通用术语。

---

## 核心术语

### 模块 (Module)
任何具有**接口**和**实现**的东西。故意跨尺度——同样适用于函数、类、包或跨层切片。
_避免_: 单元 (unit)、组件 (component)、服务 (service)

### 接口 (Interface)
调用者正确使用模块所需知道的一切。包括类型签名、不变量、顺序约束、错误模式、所需配置和性能特征。
_避免_: API、签名 (过于狭窄——仅指类型层面的表面)

### 实现 (Implementation)
模块内部——其代码体。区别于**适配器 (Adapter)**：某个东西可以是一个大实现的小适配器（如 Postgres 仓库），也可以是一个小实现的大适配器（如内存模拟）。当话题是 seam 时使用 "adapter"；其他情况使用 "implementation"。

### 深度 (Depth)
接口处的杠杆效应——调用者（或测试）每学习一个单位的接口所获得的 behaviour 量。一个模块是**深**的，当大量 behaviour 位于一个小接口后面。一个模块是**浅**的，当接口几乎和实现一样复杂。

### 接缝 (Seam) _(来自 Michael Feathers)_
可以在不原地编辑的情况下改变 behaviour 的地方。模块接口所在的位置。选择接缝放在哪里本身就是一个设计决策，与接口后面放什么不同。
_避免_: 边界 (boundary — 与 DDD 的 bounded context 重载)

### 适配器 (Adapter)
在接缝处满足接口的具体事物。描述*角色*（填补哪个槽位），而非实质（里面有什么）。

### 杠杆 (Leverage)
调用者从深度中获得的收益。每单位需要学习的接口获得更多能力。一个实现在 N 个调用点和 M 个测试中得到回报。

### 局部性 (Locality)
维护者从深度中获得的收益。变更、bug、知识和验证集中在一个地方，而不是分散在调用者之间。一次修复，到处生效。

## 核心原则

### 深度是接口的属性，不是实现的属性
一个深层模块内部可以由可 mock、可替换的小部件组成——它们只是不在接口中。模块可以有**内部接缝**（对其实现私有，由自己的测试使用）以及其接口处的**外部接缝**。

### 删除测试 (Deletion Test)
想象删除这个模块。如果复杂性消失了，模块并没有隐藏任何东西（它只是一个透传）。如果复杂性重新出现在 N 个调用者之间，那么该模块在发挥其作用。

### 接口即测试面 (Interface is Test Surface)
调用者和测试跨越同一个接缝。如果你想*绕过*接口测试，模块的形状可能是错的。

### 一个适配器 = 假设的接缝。两个适配器 = 真正的接缝。
除非某个东西实际跨接缝变化，否则不要引入接缝。

## 依赖分类 (Deepening Dependency Categories)

在评估候选深化模块时，对其依赖进行分类。类别决定了如何跨接缝测试深化后的模块：

### 1. 进程内 (In-Process)
纯计算、内存状态、无 I/O。总是可深化——合并且通过新接口直接测试。不需要适配器。

### 2. 本地可替代 (Local-Substitutable)
有本地测试替代品的依赖（PGLite 替代 Postgres，内存文件系统）。如果替代品存在则可深化。深化后的模块使用运行在测试套件中的替代品进行测试。接缝是内部的；在模块的外部接口处没有端口。

### 3. 远程但自有 (Remote but Owned) — 端口与适配器
你自己的跨网络边界的服务（微服务、内部 API）。在接缝处定义一个**端口**（接口）。深层模块拥有逻辑；传输层被注入为**适配器**。测试使用内存适配器。生产使用 HTTP/gRPC/队列适配器。

> 推荐表述: "在接缝处定义一个端口，为生产实现 HTTP 适配器，为测试实现内存适配器，使逻辑位于一个深模块中，即使它跨网络部署。"

### 4. 真正的外部 (True External) — Mock
你不控制的第三方服务（Stripe、Twilio 等）。深化后的模块将外部依赖作为注入的端口接收；测试提供 mock 适配器。

---

## 接缝纪律

- **一个适配器 = 假设的接缝。两个适配器 = 真正的接缝。** 除非至少两个适配器被证明合理（通常是生产 + 测试），否则不要引入端口。单适配器接缝只是间接层。
- **内部接缝 vs 外部接缝。** 深层模块可以有内部接缝（对其实现私有，由自己的测试使用）以及其接口处的外部接缝。不要仅仅因为测试使用内部接缝就通过接口暴露它们。

---

## 测试策略: 替换，不叠加

- 一旦深化模块的接口测试存在，浅模块的旧单元测试就变成浪费——删除它们。
- 在深化模块的接口处编写新测试。**接口即测试面**。
- 测试断言通过接口可观察的结果，而非内部状态。
- 测试应该能在内部重构时存活——它们描述的是行为，而非实现。如果测试在实现改变时必须改变，说明它在绕过接口进行测试。

## 使用场景

在以下场景中加载此参考文件：

1. **架构评审** — 当被要求评估代码库架构时，使用这些术语进行精确分析
2. **深化机会识别** — 当寻找重构机会时，应用删除测试和依赖分类
3. **设计讨论** — 当讨论模块边界、接口设计和接缝放置时
4. **测试策略** — 当讨论测试面时，使用 "接口即测试面" 原则指导决策
