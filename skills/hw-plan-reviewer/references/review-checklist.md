# ReviewChecklist: 审查清单

完整的使用 pass/fail 标准的 4 项检查清单，以及每项的具体审查程序。

---

## 检查 1: 引用验证 (Reference Verification)

验证计划中引用的所有文件、路径、行号和 API 端点是否实际存在且包含声称的内容。

### 如何验证文件引用

```
对计划中的每个文件引用:
  1. 读取该文件路径
  2. 如果读取失败 → FAIL: 文件不存在
  3. 如果读取成功 → 检查该文件是否与计划描述的用途相关
  4. 如果文件存在但与描述用途无关 → FAIL: 引用指向错误内容
```

### 如何验证行号引用

```
对计划中的每个行号引用:
  1. 读取该文件
  2. 定位到声称的行号
  3. 检查该行号附近是否包含计划描述的相关代码
  4. 如果行号存在且代码相关 → PASS
  5. 如果行号不存在或代码完全不相关 → FAIL
```

**注意:** 行号不需要精确匹配（开发者可以自行定位）；但行号范围内不应是纯空行或完全不相关的代码块。

### 如何验证"参照 X 的模式"

```
对每个 "参照 pattern/example in X":
  1. 读取 X（文件、模板、示例文件）
  2. 检查 X 是否确实展示了声称的模式
  3. 如果模式存在于 X 但实现方式略有不同 → PASS（参照，不是复制）
  4. 如果 X 中完全找不到该模式 → FAIL
```

### 如何验证 API 端点引用

```
对每个 API 端点引用:
  1. 如果端点描述包含具体路径 (如 POST /api/users)
  2. 在代码库中搜索匹配的路由定义
  3. 如果找不到路由定义且任务声称是 "新增 API" → PASS（确实需要新建）
  4. 如果找不到路由定义且任务声称是 "修改已有 API" → FAIL
```

### pass/fail 标准

| 条件 | 判定 | 理由 |
|------|------|------|
| 文件存在，内容基本相关 | **PASS** | 引用有效 |
| 文件存在，内容不太精确但有线索 | **PASS** | 开发者可以从此处探索 |
| 文件存在，但指向错误内容（完全不相关） | **FAIL** | 引用有误导性 |
| 文件不存在 | **FAIL** | 引用无效，工作无法进行 |
| 行号范围大致正确（±20 行内） | **PASS** | 允许轻微偏移 |
| 行号指向完全不相关的代码 | **FAIL** | 引用误导 |
| "参照 X 的模式"但 X 不存在 | **FAIL** | 缺少关键参考 |

## 检查 2: 可执行性检查 (Executability)

回答核心问题：开发者能否在不寻求额外澄清的情况下开始工作？

### 起点要求 (Starting Point Requirements)

每个任务的"可开始"标准：

| 信息要素 | 必要性 | 说明 |
|----------|--------|------|
| 目标文件/目录 | **必须** | 开发者要知道在哪里创建或修改代码 |
| 做什么 | **必须** | 不是"完善功能"，而是"添加 X 端点" |
| 参考/模式 | 推荐 | "参照 tasks.py 中的模式" |
| 具体行号 | 可选 | 有帮助但不是必须 |
| 完整接口签名 | 可选 | 开发者可以根据上下文自行推断 |
| 边缘情况处理 | 可选 | 不是可执行性的必要条件 |

### 依赖清晰度 (Dependency Clarity)

```
对每个任务:
  1. 如果有前置任务 → 检查前置任务是否定义在计划中
  2. 如果有外部依赖 → 检查是否说明该依赖从哪里获取
  3. 如果任务说 "参考 X 的结果" → 检查 X 是否在计划中的先执行项
```

### 工具要求 (Tool Requirements)

```
对每个任务:
  1. 如果任务说 "运行 test X" → 检查能否找到 X 的测试命令
  2. 如果任务需要特定工具 → 检查工具是常见工具还是会话环境已有
  3. 如果任务说 "部署到 Y" → 检查 Y 是否在本计划的上下文中可访问
```

### pass/fail 标准

| 条件 | 判定 | 理由 |
|------|------|------|
| 有目标文件和明确的做什么 | **PASS** | 可以开始 |
| 目标文件不明确但有明确的模式参考和做什么 | **PASS** | 从模式可推断文件 |
| 没有文件信息但有明确的搜索起点 | **PASS** | "在 auth/ 目录下添加..." 算明确 |
| 既无文件信息也无任何起点线索 | **FAIL** | 开发者不知从哪里开始 |
| 任务描述过于模糊："完善功能"、"改 bug" | **FAIL** | 完全不可开始 |
| 有前置任务但前置任务未定义 | **FAIL** | 依赖缺失 |
| 缺少工具但该工具是常见的（如 pytest, curl） | **PASS** | 环境会提供 |

## 检查 3: 关键阻塞 (Critical Blockers)

仅检查会**完全停止工作**的问题。不检查"可以使更好"的问题。

### 矛盾检测模式 (Contradiction Detection Patterns)

```
对比计划各部分:
  1. 任务 1 说"创建 auth/login.ts" → 任务 3 说"修改 auth/login.py"
     → FAIL: 文件路径矛盾（.ts vs .py）
  2. 任务 2 说"使用 REST API" → 任务 4 说"使用 gRPC"
     → FAIL: 技术方案矛盾（除非有明确的过渡说明）
  3. 设计决定说"数据库用 PostgreSQL" → 任务说"使用 MongoDB 集合"
     → FAIL: 基础设施矛盾
  4. 前言说"重构现有代码" → 所有任务都是"创建新文件"
     → 需要检查：是重构还是新建？如果是混合，需要明确哪些是重构哪些是新建
```

### 缺失信息严重性分类 (Missing Information Severity Classification)

| 缺失信息 | 严重性 | 是否阻塞 |
|----------|--------|----------|
| 不知道任务的具体文件名 | **阻塞** | 开发者找不到位置 |
| 不知道方法的参数签名 | **非阻塞** | 开发者可根据上下文推断 |
| 不知道要调用哪个 API 端点 | **阻塞** | 除非任务明确"新建 API" |
| 不知道错误处理策略 | **非阻塞** | 开发者自行处理 |
| 不知道 class/function 的完整数据结构 | **非阻塞** | 可在实现过程中定义 |
| 两个任务说的是同一个功能但描述相反 | **阻塞** | 矛盾导致无法决策 |
| 任务引用了计划中不存在的前置任务 | **阻塞** | 依赖断裂 |
| 任务的验收标准不完整 | **非阻塞** | 除非完全没有验收标准 |

### pass/fail 标准

| 条件 | 判定 |
|------|------|
| 各任务之间无矛盾 | **PASS** |
| 所有依赖链完整（前置任务存在） | **PASS** |
| 所有必须信息都存在（文件位置、做什么） | **PASS** |
| 计划中存在矛盾的指令 | **FAIL** |
| 关键依赖缺失（前置任务不存在） | **FAIL** |
| 任务完全缺少必要信息且无推断起点 | **FAIL** |

## 检查 4: QA 场景可执行性 (QA Scenario Executability)

验证每个任务是否有可验证的 QA 场景。详见 `references/qa-scenario-check.md`（加载该参考获取完整决策逻辑）。

### 快速检查标准

QA 场景可执行的**最低要求**：

```
一个可执行的 QA 场景必须包含:
  1. 具体工具 (Specific Tool) — 如 curl, pytest, go test, npm test, #selector
  2. 具体步骤 (Concrete Steps) — 分步操作，非泛泛而谈
  3. 预期结果 (Expected Results) — 可验证的断言，非主观评价
```

### pass/fail 标准

| 条件 | 判定 |
|------|------|
| 每个任务至少有 1 个 QA 场景满足上述 3 要素 | **PASS** |
| 大多数任务有可执行 QA 场景，个别任务缺少 | **REJECT**（列出缺少 QA 场景的任务） |
| QA 场景出现但不可执行（"手动检查"、"确认无误"） | **REJECT**（每个不可执行场景都计为一个阻塞问题） |
| QA 场景只有工具但无具体步骤和预期 | **REJECT** |
| QA 场景存在于任务的"验收标准"中且满足 3 要素 | **PASS** |

---

## 审查执行顺序

推荐的审查步骤（按重要性降序）：

```
第 1 步: 验证引用 → 最快失败，路径问题立即可见
第 2 步: 检查可执行性 → 读计划就能评估，不需要代码库
第 3 步: 检查关键阻塞 → 深层阅读计划，对照上下文
第 4 步: 检查 QA 场景 → 汇总检查，发现缺失模式
```

## Verification

审查提交前确认:
- [ ] 所有 FAIL 项都标记为具体问题（精确文件路径/任务名）
- [ ] 没有将"非阻塞"项列为问题（参看 `references/anti-patterns.md`）
- [ ] 问题总数不超过 3 个（如果超过，只保留最关键的 3 个）
- [ ] 每个问题都有具体建议（怎么改，而不仅仅是"不够好"）
