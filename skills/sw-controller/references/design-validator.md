# 设计多模型交叉验证 (Multi-Model Design Validator)

参考: Compound Engineering heterogeneous review (异质审查者并行审查) +
      BMAD adversarial review (对抗性审查 — 不同的 "思维模式" 发现不同类型的问题)

## 核心理念

单一模型审查设计文档存在**系统性盲区**——模型在训练数据、RLHF 偏好、推理链路上的相似性，会导致同类型问题被反复遗漏。这不是审查维度不够多的问题，而是审查者 "脑回路" 太相似的问题。

多模型交叉验证解决的是:
- **盲区互补:** 模型 A 忽略的问题，模型 B 可能抓到
- **信度校准:** 多个模型都发现的问题 → 高置信度；只有一个模型发现 → 可能是误报
- **对抗性检验:** 让不同模型相互 "挑刺"，暴露单一模型审查不会触及的深层矛盾

## 何时启用

验证器是**可选层**，不是每个项目都需要。由 `business_domain` 驱动:

| business_domain | 验证器 | 最少模型数 | 说明 |
|-----------------|--------|-----------|------|
| `fintech` | **强制开启** | 3 | 资金安全 + 合规要求，系统盲区不可接受 |
| `ecommerce` | **默认开启** | 2 | 支付 + 用户数据，建议开启但可手动关闭 |
| `general` | **可选** | 2 | 根据需求风险等级决定 |
| `internal-tools` | **默认关闭** | — | 内部工具风险可控，投入产出比不高 |

可通过 `_context/config.yaml` 覆盖:

```yaml
sw:
  multi_model_validation:
    enabled: true/false        # 全局开关，覆盖 business_domain 默认值
    min_models: 2              # 最少参与验证的模型数 (2-4)
    max_models: 3              # 最多参与验证的模型数 (控制成本)
    cost_budget_usd: 2.00      # 单次验证最大预算 ($)
    consensus_threshold: 0.5   # 共识阈值 — 超过此比例的模型都发现问题 = 高置信度
```

## 模型选择策略

不绑定具体模型名称（平台无关）。用**能力画像**描述验证者:

### 验证者画像 (Validator Profiles)

| 画像 | 推理风格 | 擅长发现 | 盲区 |
|------|---------|---------|------|
| **分析型 (Analyst)** | 深度推理、链式思维、穷举边界 | 逻辑矛盾、状态机遗漏、边界条件、数据一致性 | 可能在细节上过度投资，忽略宏观结构问题 |
| **直觉型 (Scanner)** | 快速模式匹配、广覆盖、经验驱动 | 反模式、安全漏洞、常见陷阱、合规缺口 | 可能产生较多假阳性，对新颖架构判断不准 |
| **对抗型 (Adversary)** | 故意挑错、最坏情况推演、压力测试 | 单点故障、安全漏洞、降级/回滚缺失、假设错误 | 可能过于悲观，对低概率风险过度反应 |
| **系统型 (Architect)** | 全局视角、跨组件依赖、演进思维 | 架构矛盾、耦合问题、扩展性瓶颈、ADR 一致性 | 对实现细节不够敏感 |

### 画像组合推荐

| 模型数 | 组合 | 覆盖的盲区 |
|--------|------|-----------|
| 2 | 分析型 + 对抗型 | 逻辑矛盾 + 安全隐患 |
| 3 | 分析型 + 对抗型 + 直觉型 | 全面覆盖（推荐，fintech/ecommerce 标准） |
| 4 | 全画像 | 最大覆盖，仅关键项目，成本最高 |

**模型分配规则:**
- 如果可用的 API 模型 ≥ 3 个且画像不同 → 使用 3 模型组合
- 如果可用的 API 模型 = 2 个 → 使用分析型 + 对抗型
- 如果只有 1 个模型可用 → 降级为单模型深度审查（不是交叉验证，但仍比无验证好）
- 永远不要用同一个模型的两次调用来冒充 "两个验证者"

**平台模型映射指引（非约束）:**
- 分析型画像适合分配: Claude Opus, GPT-4o, DeepSeek-V3
- 直觉型画像适合分配: Claude Haiku, GPT-4o-mini, Gemini Flash
- 对抗型画像适合分配: Claude Sonnet, GPT-4o (system prompt 加对抗指令)
- 系统型画像适合分配: Claude Opus, GPT-4o (system prompt 加架构审查指令)

## 验证流程 (3 步)

### 第 1 步: 验证准备 (Validation Setup)

**输入:**
- 设计文档: `designs/{id}-design.md`
- 关联的需求规格: `requirements/{id}.md`
- 头脑风暴记录: `designs/{id}-brainstorm.md`（如有）
- 知识库 ADR: `knowledge-base/decisions/ADR-*.md`

**准备动作:**

1. **读取配置:** 从 `_context/config.yaml` 读取 `multi_model_validation` 配置
2. **确定画像组合:** 根据 `max_models` 和可用模型选择画像
3. **分配验证指令:** 为每个选定的画像生成验证提示词

**验证提示词模板（按画像）:**

```
分析型 (Analyst) 指令:
  "你是一个系统分析师。请审查以下设计文档，重点关注:
   1. 逻辑一致性 — 组件间的契约是否自洽？
   2. 完整性 — 是否遗漏了关键状态转换或错误路径？
   3. 边界条件 — 并发、空值、超时、重入场景是否被处理？
   4. 数据一致性 — 分布式事务/补偿策略是否合理？
   对每个发现，标注置信度 (High/Medium/Low) 和建议优先级 (P0/P1/P2/P3)。"

直觉型 (Scanner) 指令:
  "你是一个代码审查专家，擅长快速发现反模式和常见陷阱。请扫描以下设计文档，寻找:
   1. 已知反模式 (N+1 查询、上帝对象、循环依赖)
   2. 安全漏洞 (注入、XSS、越权、敏感数据泄露)
   3. 常见遗漏 (缺少速率限制、缺少幂等键、缺少审计日志)
   4. 配置/部署陷阱 (硬编码值、缺少健康检查)
   对每个发现，标注置信度 (High/Medium/Low) 和建议优先级 (P0/P1/P2/P3)。"

对抗型 (Adversary) 指令:
  "你是一个质量保障工程师，你的任务是「破坏」这个设计。审查设计文档，推演:
   1. 最可能出错的场景是什么？
   2. 如果某个外部依赖宕机，会发生什么？
   3. 如果流量突然 10x，哪些组件会先崩溃？
   4. 恶意用户会怎么利用这个系统？
   对每个发现，标注置信度 (High/Medium/Low) 和建议优先级 (P0/P1/P2/P3)。"

系统型 (Architect) 指令:
  "你是一个架构师。请审查以下设计文档，关注:
   1. 与已有 ADR 的一致性 — 是否违反已建立的架构原则？
   2. 组件耦合度 — 变更一个组件是否会导致级联修改？
   3. 演进空间 — 这个设计是否锁死了未来的选择？
   4. 可测试性 — 设计是否易于单元测试和集成测试？
   对每个发现，标注置信度 (High/Medium/Low) 和建议优先级 (P0/P1/P2/P3)。"
```

4. **成本预估:** 基于设计文档 token 数 × 模型数量 × 输出 token 预估，检查是否超过 `cost_budget_usd`。如果超了 → 报告人类并建议:
   - (a) 减少模型数量
   - (b) 用更小的模型替代
   - (c) 提高预算上限
   - (d) 跳过验证（风险自负）

### 第 2 步: 并行验证执行 (Parallel Validation)

**执行规则:**
- 所有选定的验证者**同时启动**（互不依赖）
- 每个验证者拿到: 设计文档路径 + 对应的画像指令
- 验证者不相互通信——独立的 "第一次阅读" 才能暴露盲区
- 每个验证者输出结构化发现列表

**单验证者输出格式:**

```json
{
  "validator_profile": "analyst",
  "model": "{实际使用的模型名称}",
  "findings": [
    {
      "id": "F-{profile}-{NN}",
      "severity": "P0|P1|P2|P3",
      "category": "logic|security|performance|consistency|completeness|ambiguity",
      "location": "设计文档 {section} 节, 决策 D-{N}",
      "description": "{问题描述}",
      "impact": "{如果不修复会怎样}",
      "suggestion": "{修复建议}",
      "confidence": "high|medium|low"
    }
  ],
  "positive_observations": [
    "{设计中做得好的地方 — 不是只找问题}"
  ],
  "token_consumed": {
    "input": 12345,
    "output": 2345
  }
}
```

### 第 3 步: 发现聚合与共识分析 (Aggregation & Consensus)

**目标:** 把多个验证者的独立发现合并成一个统一的、按置信度加权的发现列表。

**聚合算法:**

```
1. 语义去重 (Semantic Dedup):
   比较所有发现的 description + location。
   如果两个验证者发现了"同一位置 + 同一问题类型" →
     合并为一个发现，标注为 "多验证者确认"
   如果只有部分验证者发现了 →
     标注为 "部分验证者确认"
   如果只有一个验证者发现 →
     标注为 "单验证者发现"

2. 置信度计算:
   多验证者确认 (所有验证者)     → 置信度: HIGH   → 很可能真实
   多验证者确认 (多数验证者)     → 置信度: MEDIUM → 需要关注
   单验证者发现 (high individual) → 置信度: LOW    → 可能是误报，检查
   单验证者发现 (low individual)  → 置信度: LOW    → 很可能是误报，标记但不阻塞

3. 分歧检测:
   如果两个验证者给出了相反的建议 →
     标记为 "MODEL_DISAGREEMENT"
     需要人工裁决 — 不自动选择
```

**聚合结果表:**

| ID | 严重度 | 类别 | 问题描述 | 验证者共识 | 置信度 | 需要人工? |
|----|--------|------|---------|-----------|--------|----------|
| F-001 | P0 | security | {描述} | Analyst + Adversary + Scanner | HIGH | 否 |
| F-002 | P1 | completeness | {描述} | Adversary only | LOW | 是(确认) |
| F-003 | P1 | consistency | {描述} | Analyst → 方案A / Scanner → 方案B | DISAGREEMENT | 是(裁决) |

**自动决策规则:**
- HIGH 置信度 + P0/P1 → **自动通过**，直接标记为需修复
- HIGH 置信度 + P2/P3 → **自动通过**，加入修复列表
- LOW 置信度 + P0 → **标记为需人工确认**（可能是误报，但风险太高不敢忽略）
- LOW 置信度 + P1/P2 → **标记为建议修复**，不阻塞门禁
- LOW 置信度 + P3 → **自动归档**，不阻塞
- DISAGREEMENT → **升级给人类裁决**

## 成本监控

每次验证运行后，更新累计成本:

```yaml
# _context/memory/sw-controller/cost-tracking.yaml
validation_runs:
  - design_id: "DESIGN-20260501-001"
    timestamp: "2026-05-01T15:30:00Z"
    models_used: ["claude-sonnet-4-6", "claude-opus-4-6", "claude-haiku-4-5"]
    total_cost_usd: 1.47
    findings: {P0: 1, P1: 3, P2: 5, P3: 8}
    consensus_rate: 0.62  # 62% 的发现被 ≥2 个模型确认
```

成本告警阈值（与全局 API 预算联动）:
- 单次验证成本 > `cost_budget_usd` → 立即停止，报告人类
- 累计日成本 > 日预算的 30% → 警告（后续验证降级到 2 模型）

## 向设计门禁的传递

验证报告生成后:

1. **将 HIGH 置信度的 P0/P1 发现** 写入设计文档的开放问题或修正章节
2. **将分歧记录** 提交给人类裁决
3. **更新设计门禁检查** — 在 `design-gate.md` 的 G1 检查中增加:
   - [ ] 多模型验证完成（如启用）
   - [ ] HIGH 置信度 P0/P1 已处理
   - [ ] 模型分歧已由人类裁决

## 降级策略

以下情况自动降级为单模型深度审查（不是多模型交叉验证）:

| 降级原因 | 处理方式 |
|---------|---------|
| 可用模型 < 2 个 | 单模型审查 + 在报告中标注 "单模型验证，置信度无法校准" |
| 设计文档 < 500 tokens | 跳过验证（文档太小，交叉验证意义不大） |
| 预估成本 > budget | 报告人类 → 选 (a) 减少模型 (b) 提高预算 (c) 跳过 |
| API 配额不足 | 降级为最小模型单次审查 |
| `internal-tools` 且未手动开启 | 跳过验证 |

降级时在验证报告开头明确标注:

> ⚠️ 验证降级: 由于 {原因}，本次验证降级为 {处理方式}。交叉验证的置信度校准和盲区互补功能未生效。
