# 失败恢复协议 (Failure Recovery)

参考: Sisyphus orchestrator DNA — oh-my-openagent Phase 2C 失败恢复

## 核心理念

失败是开发过程中的正常现象，但**重复同样的失败是不可接受的**。失败恢复协议确保每次失败都转化为学习，而不是简单的重试循环。

**铁律:**
- 修复根因，不治症状
- 每次修复后必须重新验证
- 3 次连续失败 → 必须停下来
- 永远不要让代码停留在 broken 状态
- 绝不删除失败的测试来 "修复" 构建

## 3 次连续失败规则

这是失败恢复协议中最关键的安全阀：

```
同一条问题连续失败 3 次:
  STOP    → 停止一切修复尝试
  REVERT  → 回退所有变更到最后一个已知 good 状态
  DOCUMENT → 记录: 尝试了什么、为什么失败、学到了什么
  CONSULT → 委托 hw-strategic-advisor 分析
  ASK USER → 简洁的问题陈述 + 可选方案，等待用户决策
```

### 何谓 "同一条问题"？

同一条问题 = 同一个测试用例失败 / 同一个 lint 错误 / 同一个编译错误。

```
示例 1 — 同一条问题，触发 3 次规则:
  尝试 1: 修改 UserService.java → 编译失败: symbol not found
  尝试 2: 调整 import → 仍然编译失败: symbol not found
  尝试 3: 修改方法签名 → 仍然编译失败: symbol not found
  → 触发 STOP → REVERT → DOCUMENT → CONSULT → ASK USER

示例 2 — 不同问题，不触发:
  尝试 1: UT 失败 → 修复业务逻辑
  尝试 2: API 测试失败 → 修复 controller 参数绑定
  尝试 3: 安全检查报告 P0 → 修复 SQL 注入
  → 这是 3 个不同的问题，不触发
```

### 同一种方法不同实例，也触发

如果连续 3 次用同一种策略去修同一个问题（即使每次改的代码不同），也触发：

```
示例:
  问题: NullPointerException in UserService.getUser()
  尝试 1: 加 null check → 仍然报错（另一个 NPE）
  尝试 2: 再加一个 null check → 仍然报错（又一个 NPE）
  尝试 3: 再加 null check → 仍然报错
  → 触发！策略是 "加 null check"，应该在第一次失败后考虑: 为什么会出现 null？
     可能是上游数据源问题，可能是设计问题。
```

## 根因分析模板

每次失败后，在动手修复前，先做根因分析：

```markdown
### 失败分析

**现象:** {测试名称/错误信息/堆栈摘要}
**直接原因:** {什么代码导致了此错误}
**根因:** {为什么这段代码会有问题}

**根因分类:**
- [ ] 逻辑错误: 代码逻辑本身有 bug
- [ ] 假设错误: 对环境/输入/状态的假设不成立
- [ ] 设计缺陷: 架构层面的问题导致此类错误不可避免
- [ ] 环境问题: 配置/依赖/基础设施导致
- [ ] 测试问题: 测试本身写错了
- [ ] 信息缺失: 因为不了解某些上下文导致

**修复策略:** {只填一项，填入后再执行}
- [ ] 最小修复 (改 1-2 行)
- [ ] 局部重构 (改同一文件内的多处)
- [ ] 设计调整 (改多个文件)
- [ ] 回退并重新设计
```

### 根因分析的关键问题

在判定根因之前，问自己这 5 个问题：

1. **为什么这个错误现在才出现？** — 是新增代码引入的，还是之前就存在？
2. **修复这个错误是否会破坏其他东西？** — 这个常量/方法/类型还被谁依赖？
3. **类似的错误是否可能存在于其他地方？** — 如果是模式问题，全局搜索
4. **这个错误在什么条件下触发？** — 边界情况还是常规路径？
5. **测试是否正确地表达了意图？** — 有时错误在测试，不在代码

## 重新验证检查清单

每次修复后，必须执行以下验证才认为修复完成：

- [ ] 原始失败已消失（测试 PASS / 编译通过 / lint 无错误）
- [ ] 没有引入新的失败（全量测试，不只是修的那个）
- [ ] 修复不破坏现有功能（相关模块的测试仍然 PASS）
- [ ] 修复不引入安全/性能回归（如果修复涉及关键路径）
- [ ] 修复不违反代码规范（linter 通过）
- [ ] 修复是合理的（不是 hack，不是 copy-paste，不是临时绕过）

## STOP 后的完整流程

### 第 1 步: STOP

立即停止所有修复尝试。不要 "再试一次"，不要 "换个方式试试"。3 次失败是一个硬性阈值。

### 第 2 步: REVERT

回退到最后一个已知 good 状态：

```
回退策略:
  1. 如果有 git commit → git reset --hard HEAD~{n} (回退 n 次失败的 commit)
  2. 如果没有 commit → 手动回退所有变更的文件
  3. 确认回退后: 编译通过 + 测试通过 (原始状态)
```

**重要:** 回退时记录回退了什么。这有助于后续分析。

```markdown
### 回退记录

**回退的文件:**
- src/service/UserService.java: {回退了什么改动}
- src/controller/UserController.java: {回退了什么改动}

**回退后的状态:**
- 编译: ✅
- 测试: ✅ (原始测试全部 PASS)
- Lint: ✅
```

### 第 3 步: DOCUMENT

记录尝试过的方案和失败原因。这是知识积累的关键一步——防止以后重复同样的错误。

```markdown
### 失败尝试记录

**问题:** {简要描述}
**尝试 1:** {方案} → 失败原因: {为什么失败了}
**尝试 2:** {方案} → 失败原因: {为什么失败了}
**尝试 3:** {方案} → 失败原因: {为什么失败了}

**模式识别:**
- 共同失败点: {3 次尝试的共同问题}
- 可能的根因: {根因假设}

**学到的东西:** {这次失败教会了我们什么}
```

### 第 4 步: CONSULT

委托 hw-strategic-advisor 做深度分析：

```
TASK: 分析 {问题描述} 的 3 次失败尝试
CONTEXT: 提供失败尝试记录
EXPECTED OUTCOME: 根因假设 + 替代方案建议
MUST DO: 解释为什么之前的方案都失败了
MUST NOT DO: 建议与已尝试方案类似的方法
```

### 第 5 步: ASK USER

向用户呈现简洁的问题陈述 + 方案选项：

```
格式:

**问题:** {1 句话描述}
**已尝试方案:** {简述 3 次尝试}
**咨询建议:** {strategic-advisor 的建议}

**选项:**
A. {方案 A — 具体可操作}
B. {方案 B — 具体可操作}
C. 进一步调查: {需要什么信息}
D. 其他: {用户自定义方案}

**推荐:** {推荐哪个选项，为什么}

等待你的决定。
```

## 禁止行为 (绝不)

### 1. 绝不将代码留在 broken 状态

```
❌ 错误: "测试失败，但我知道问题在哪，先提交让 CI 跑一下"
✅ 正确: 修复 → 本地测试 PASS → 提交

❌ 错误: 修复一半下班/切换任务
✅ 正确: 如果无法完成 → REVERT 到 good 状态 → 下次继续
```

### 2. 绝不删除失败的测试来 "修复" 构建

```
❌ 错误: 删除或 skip 失败的测试
✅ 正确: 修复代码让测试 PASS，或确认测试本身有误后修改测试

❌ 错误: 把 assert 改成宽松的比较
✅ 正确: 理解测试的原始意图，确认预期值为什么不对
```

### 3. 绝不重复同样的策略

```
❌ 错误: 错误是 NullPointerException → 一直加 null check
✅ 正确: 第 1 次 null check → 如果仍失败 → 分析为什么会有 null
                                   → 第 2 次从源头解决 null
```

### 4. 绝不在没有验证的情况下声称 "修复了"

```
❌ 错误: "应该没问题了" → 没有运行测试就标记完成
✅ 正确: 运行了相关测试 → 确认 PASS → 标记完成
```

## 升级到人工的条件总结

| 条件 | 动作 |
|------|------|
| 同一问题连续失败 3 次 | STOP → REVERT → DOCUMENT → CONSULT → ASK USER |
| 修复引入了新的 P0 问题 | 立即 REVERT 该修复 → ASK USER |
| 修复需要在多个不熟悉的模块间改动 | 先 CONSULT strategic-advisor → 再决定 |
| 问题涉及数据安全/资金/合规 | 立即升级，不等待 3 次尝试 |
| 根因是设计层面的缺陷 | 升级，建议回到设计阶段 |

## 与门禁系统的关系

失败恢复协议与质量门禁 (`quality-gates.md`) 协同工作：

```
GATE 1 (TDD) 失败 → 适用此协议 (3 次尝试)
GATE 2 (审查) P0/P1 → 适用此协议 (3 次尝试)
GATE 3 (回归) 失败 → 适用此协议 (回退 + 定位破坏者)
```

### 自动修复次数 (来自 quality-gates.md)

| 严重度 | 最大自动尝试次数 | 超限后 |
|--------|----------------|--------|
| P0 | 3 次 | STOP → 升级人工 |
| P1 | 3 次 | STOP → 升级人工 |
| P2 | 2 次 | 升级人工 |
| P3 | 不自动修复 | — |
```
